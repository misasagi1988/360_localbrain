# 本脑各版本SAE规则配置说明

标签（空格分隔）： SAE_DOC

---

本文档基于本脑2.0 sim2.0版本

## 1. 规则模板说明

当前SAE规则共支持十一个模板，具体说明如下:

| 模板名称                       | 模板描述                                                     | 事件源配置                           |
| ------------------------------ | ------------------------------------------------------------ | ------------------------------------ |
| 普通模板                       | 基于单个事件的触发，一对一告警。                             | 有且只有一个事件源                   |
| 普通模板-having distinct count | 在一段时间内，相同分组内多个事件中某个属性值不同的次数满足条件即触发告警。统计类告警，分组字段非必须。 | 有且只有一个事件源                   |
| 普通模板-having count          | 在一段时间内，相同分组内某事件发生的次数满足条件即触发告警，如果配置了distinct条件，则需要同时满足相同分组内多个事件中该字段属性值不同的次数满足条件才可触发告警。统计类告警，分组字段非必须。 | 有且只有一个事件源                   |
| 普通模板-having sum            | 在一段时间内，相同分组内多个事件中某属性值的和满足条件即触发告警。统计类告警，分组字段非必须。 | 有且只有一个事件源                   |
| 关联模板-follow by             | 某事件A发生之后的一段时间内发生了事件B，关联类告警，关联条件非必须，**涉及3个及以上事件时，关联条件只支持与逻辑**。 | 至少两个事件源，最多支持五个事件源。 |
| 关联模板-not follow by         | 某事件A发生之后的一段时间内一定不发生事件B，关联类告警，关联条件非必须。 | 有且只有两个事件源                   |
| 关联模板-or follow by          | 在一段时间内，事件A和事件B均发生，但对发生时间无先后顺序要求，关联类告警，关联条件非必须。 | 有且只有两个事件源                   |
| 关联模板-repeat until          | 至少M次事件A发生之后的一段时间内发生了事件B，关联类告警，关联条件非必须。 | 有且只有两个事件源                   |
| 关联模板-not before            | 事件B发生之前的一段时间内没有发生事件A，关联类告警。关联条件非必须，如配置，**要求关联条件只支持多个与逻辑运算，且A和B的关联字段数量必须一致**。 | 有且只有两个事件源                   |
| 关联模板-any order             | 一段时间内发生多个事件，发生时间无顺序要求，关联类告警。**不允许配置关联条件，可配置分组条件**。 | 至少两个事件源，最多支持五个事件源。 |
| 普通模板-not occur             | 某个事件曾经发生过，后续一段时间内没有再次发生。**可配置分组条件**。 | 有且只有一个事件源                   |

## 2. 事件源说明

### 2.1 事件名称

事件源中可以使用的事件名称分为以下几种，全局事件需配置过滤条件。具体说明如下：

| 事件                   | 说明                                                         |
| ---------------------- | ------------------------------------------------------------ |
| 普通事件               | dv解析后生成的事件                                           |
| 内部事件               | 类似告警，由SAE生成，可供其他规则使用，以构造更复杂的场景。  |
| 威胁情报IP匹配事件     | 提取dv解析后的事件中的源地址和源端口，目的地址和目的端口，URL中的主机IP，分别与当前环境中的威胁情报信息做比较，如果匹配，则生成该事件。 如果事件中的源地址、目的地址或URL中的主机IP匹配威胁情报，都会生成该事件，分别对应源地址匹配、目的地址匹配和URL匹配。 |
| 威胁情报Domain匹配事件 | 提取dv解析后的事件中的域名，URL中的主机域名，与当前环境中的威胁情报信息做比较，如果匹配，则生成该事件 如果事件中的域名或URL中的主机域名匹配威胁情报，都会生成该事件，分别对应域名匹配、URL匹配。 |
| 威胁情报URL匹配事件    | 提取dv解析后的事件中的URL，与当前环境中的威胁情报信息做比较，如果匹配，则生成该事件，对应URL匹配。 |
| 漏洞利用匹配事件       | 提取dv解析后的事件中的目的地址，与当前环境中的资产脆弱性信息做比较，如果匹配，则生成该事件 |
| 全局事件               | 包含以上任意事件                                             |

### 2.2 事件源引用对象与属性

在事件源中，可以配置事件使用的基础对象，对象支持多选或不选。

| 对象         | 对象属性字段说明                                             |
| ------------ | ------------------------------------------------------------ |
| 基础对象     | 字段可用于过滤条件、关联条件、时间窗口、输出字段配置等       |
| 情报类型     | 用于多维度关联分析，字段**仅可用于过滤条件、关联条件配置**，名称以"情报类型."开头 |
| 主机资产类型 | 用于多维度关联分析，字段**仅可用于过滤条件、关联条件配置**，名称以"资产.源地址."开头的用于使用源地址与内部资产匹配，名称以"资产.目的地址."开头的用于使用目的地址与内部资产匹配 |
| 脆弱性类型   | 用于多维度关联分析，字段**仅可用于过滤条件、关联条件配置**，名称以"脆弱性类型."开头 |

当前系统支持的属性类型有IP(ipv4)/整形/长整型/浮点数/字符/时间/枚举这几种，属性字段可为数组。

## 3. 操作符说明

### 3.1 系统支持的操作符

当前系统支持的操作符有=/!=/>/>=/</<=/exist/not_exist/like/plike/rlike/in/belong/contain/match，左操作数需配置为事件的具体属性，右操作数既支持使用录入数据(V)，也支持使用关联字段(F)，过滤条件也支持右操作数为字段。

| 操作符             | 使用说明                                                                                                                                                                                                                                                                                                                                                   |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| =/!=            | 关系操作符，支持任意类型属性，包括数组，可用于过滤条件和关联条件。                                                                                                                                                                                                                                                                                                                      |
| >/>=/</<=       | 关系操作符，支持除数组以外的任意类型属性，可用于过滤条件和关联条件。                                                                                                                                                                                                                                                                                                                     |
| exist/not_exist | 存在/不存在，无右操作数，支持任意类型属性，可用于过滤条件。                                                                                                                                                                                                                                                                                                                         |
| like            | 字符串包含，支持字符类型属性和字符串数组类型属性，对于字符串数组类型，只要数组的任意元素包含该字符串即可匹配，**忽略大小写匹配**。可用于过滤条件和关联条件。                                                                                                                                                                                                                                                                       |
| plike           | 字符串起始包含，支持字符类型属性和字符串数组类型属性，对于字符串数组类型，只要数组的任意元素包含该字符串即可匹配，**忽略大小写匹配**。可用于过滤条件和关联条件。                                                                                                                                                                                                                                                                     |
| rlike           | **正则全匹配，忽略大小写，.支持匹配换行符**，仅支持字符类型属性，特殊字符需转义。对规则处理性能影响较大，配置时注意后移。可用于过滤条件。                                                                                                                                                                                                                                                                                |
| in              | 支持IP及数组格式字符串 IP：左操作数属于网段；数组格式字符串：数组的元素包含左操作数。可用于过滤条件。，4.0开始，in                                                                                                                                                                                                                                                                                                  |
| belong          | 属于特征，右操作数仅支持F，用于匹配安全信息和动态信息。 安全信息中，字符类信息细分为**字符串比较/正则全匹配/正则部分匹配**三种子类型： 字符串比较是用事件的字段值与该信息组的内容做比较，只要该事件的字段值与该信息组的任一内容相等即可(**大小写敏感**)； 正则全匹配类型是用事件的字段值与该信息组的内容做正则模式全匹配，只要该事件的字段值能够完全匹配该信息组的任一正则表达式即可(**忽略大小写**)； 正则部分匹配类型是用事件的字段值与该信息组的内容做正则模式部分匹配，只要该事件的字段值能够部分匹配该信息组的任一正则表达式即可(**忽略大小写**)。 **正则类安全信息，.支持匹配换行符**。正则类、ip类信息组对规则处理性能影响较大，配置时注意后移。可用于过滤条件。 |
| contain         | 关联包含，仅支持数组类型，数组元素可为任意类型，数组任意元素内容与右值相同即可匹配，**字符类属性大小写敏感**，可用于过滤条件和关联条件。                                                                                                                                                                                                                                                                                 |
| match           | 匹配情报，无右操作数，仅针对源地址/目的地址/域名/URL字段有效。指明具体是哪个字段匹配到了情报。可用于过滤条件。                                                                                                                                                                                                                                                                                             |
| inmap           | 匹配多维信息组，多维信息组的key和value仅支持单个值，不支持区间格式。右操作数为信息组名称.字段名称格式，可用于过滤条件。demo: 源地址 inmap 多维信息组测试.目的地址，目的地址是多维信息组的键，且源地址在目的地址对应多维信息组键的值的set里。即key&value都匹配。                                                                                                                                                                                                      |
| not_inmap       | 匹配多维信息组，多维信息组的key和value仅支持单个值，不支持区间格式。右操作数为信息组名称.字段名称格式，可用于过滤条件。demo: 源地址 not_inmap 多维信息组测试.目的地址，目的地址是多维信息组的键，且源地址不在目的地址对应多维信息组键的值的set里。即key匹配，value不匹配。                                                                                                                                                                                               |

in操作符用于数组情况和contain操作符的区别：

contain：左操作数是个数组类型的字段，右操作数可以是一个输入值，或者一个非数组类型的字段，表示数组包含某个值，可用于过滤条件和关联条件；

in: 左操作数是个非数组类型的字段，右操作数是“[...]”格式的字符串，表示某个元素在数组里，仅可用于过滤条件。

目前，只有like、plike、rlike及belong正则类信息组是忽略大小写的，其他操作符及belong字符串比较类信息组都是大小写敏感的。

update from 5.0: 数组类型的属性可以支持的操作符列表:  =/!=/in/exist/not_exist/contain/like/plike/rlike; 非数组类型的属性不支持的操作符列表: contain
数组字段 `=` 操作符转换为 `contain` 操作符
数组字段 `!=` 操作符转换为 `not contain` 操作符
数组字段 `in` 操作符转换为 `contain` 操作符

## 4. 多维度关联分析

可以将威胁情报信息、资产信息、漏洞数据信息融入到sae规则配置中，从威胁情报、资产、脆弱性三个维度来辅助告警逻辑判断，以有效降低误报，精准告警。多个维度可以混合使用。

威胁情报匹配事件、漏洞利用匹配事件是通过多维度关联分析来生成的。多维度关联分析匹配逻辑可参考：本脑多维度关联分析匹配逻辑文档

## 5. 配置注意点

### 5.1 模板选择

- 不同模板用途是不一样的，处理逻辑和性能也是不同的，在选择时注意其应用场景，选择合适的模板。

- 部分模板存在一些已知问题，选取时需注意：

- 统计类规则，输出字段值取最后一个事件的字段值

- not before、not occur模板规则，第一个告警可能会生成不了，存在告警漏报问题

- repeat until模板规则，配置的阈值是一个下限，只保留前面M个事件，可能因AB时间差不满足条件，存在告警漏报问题；如果规则配置了关联条件，实际只对第一个A事件和B事件的字段做了限制，存在告警误报问题

### 5.2 事件源配置

- 全局事件需配置过滤条件。

- 如果确定了该规则的事件名称，应指定事件源的事件名称，而不要采用全局事件。

- 对于威胁情报相关的规则，事件源名称应配置为具体的匹配事件。如果事件源配置为全局事件，需要在过滤条件中使用match操作符过滤具体的匹配字段。

- 对于脆弱性相关的规则，事件源名称应配置为“漏洞利用匹配事件”，当前不支持事件源配置为全局事件，而在过滤条件中指定事件名称。

### 5.3 过滤条件配置

- 注意各个子表达式的先后顺序，引擎在过滤时是有短路的，表达式的先后顺序对性能是有些影响的。

- 由于日志解析中，对于不同的数据源，协议字段可能解析成各种大小写不一致的日志，比如tcp/Tcp/TCP等，为了方便写规则，sae对这个字段做了统一转换成大写的处理，安全同事在配置规则时需要注意这个字段。右操作数统一为大写，字符串比较类信息组中的值也需要配置为大写。比如可以统一写成: 协议="TCP"。

### 5.4 时间窗口配置

- 时间窗口的定义：从第一笔满足条件的事件进入引擎开始，最大允许的时间范围。在该范围内只要数据满足条件，即可触发告警，并将状态清零。

### 5.5 输出属性与内部事件

- 内部事件类似于告警，它的属性只有生成该内部事件的规则所配置的所有输出字段，生成时注意输出字段选取。当前我们规则中有些内部事件的输出字段不完整，导致引用该内部事件做再次匹配的规则因字段缺失而无效。

- 内部事件的事件名称建议有统一格式。

- 普通模板规则不建议只生成内部事件。

### 5.6 动态信息配置

- 动态信息组：类似系统的安全信息配置，可以在过滤条件中使用belong操作符引用。它的优点是支持配置信息组过期时间，可以根据规则配置，在告警使能时，将某个字段的值加入动态信息，实现信息组的动态变化。

### 5.7 其他注意点

- 除not occur模板规则外，其他模板规则必须配置输出字段。

- 规则名称/告警内容/告警处置建议支持采用$引用输出字段。

- 标记属性字段，相当于给告警打了一个标记，不允许与输出属性重复。

- 尽量精简规则数量及内部事件生成量，保证其通用性，以保证引擎处理性能。

- 安全信息：当前系统中内置安全信息分为IP/数字/字符/时间四种类型，在配置时注意尽量精简信息组内容的数量，保证处理性能。

## 6. 安全事件合并策略和告警合并策略说明

事件合并策略(direction_key)相同的告警，会被合入同一个安全事件。

告警合并策略(merge_key)相同的告警，会被合入同一个合并告警，合并告警按天合并。

这两个字段都是由多个字段通过下划线拼接而成。当前系统共有EDR/NDR/XDR三种场景模型，不同场景模型下，告警的merge_key及direction_key使用的字段不同，合并告警所有的字段也不同，输出属性中注意配置，没有则为NULL。EDR场景类安全事件可绘制进程树，NDR/XDR场景类安全事件可绘制攻击链路图。

- EDR场景：

- direction_key: client_host_sign, local_date(本地日期)，客户端标识相同的告警，按自然天合入同一个安全事件

- merge_key: id，由于告警id唯一，告警不合并，每个告警单独生成一个合并告警

- NDR/XDR场景：

- direction_key: sae_rule_id, alarm_name(哈希值), internal_direction_key，同一规则、告警名称相同、攻击方向相同的告警会合入同一个安全事件。

- 攻击方向计算方法：

- 如果攻击方向有多个，攻击方向取值为其他

- 源地址组中有内网IP，目的地址组中有外网IP，则是内对外的告警，值为${源地址}_0

- 如果不是内对外的告警，则判断源地址组中有内网IP，目的地址组中也有内网IP，是内对内的告警，值为${源地址}_0

- 如果不是内对内的告警，则判断源地址组中有外网IP，目的地址组中也有内网IP，是外对内的告警，值为0_${目的地址}

- 如果都不是，则为外对外的告警，值为0_0

- merge_key: src_address, dst_address, alarm_content(哈希值)

本脑2.1版本默认merge_key引用字段有变化：

EDR场景merge_key: host_ip, alarm_name, alarm_content

NDR场景merge_key: src_address,dst_address, alarm_name, alarm_content

==**update:**==
本脑4.0版本以后，告警攻击方向计算方法：  
- 获取告警关联的每一笔原始日志的源地址和目的地址，判断每笔日志的源地址和目的地址是否为内网IP，计算每笔日志的攻击方向:  
  - 如果源地址和目的地址都不存在，或只存在一个，则攻击方向为其他  
  - 如果源地址和目的地址都是内网IP，则攻击方向为内对内  
  - 如果源地址是内网IP，目的地址是外网IP，则攻击方向为内对外  
  - 如果源地址是外网IP，目的地址是内网IP，则攻击方向为外对内  
  - 如果源地址是外网IP，目的地址是外网IP，则攻击方向为其他  
- 告警攻击方向根据告警关联的原始日志计算得出，如果告警关联的原始日志中存在多个攻击方向，则攻击方向取值为其他，如果攻击方向只有一个，则攻击方向取值为该攻击方向
