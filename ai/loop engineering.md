 # 什么是 Loop Engineering

  简单说就是让 AI 自己写完 → 自己检查 → 自己改 → 循环，而不是你手动反复说"这里不对改一下"。

  基本模式：

  普通模式：你写代码 → 你发现报错 → 你告诉 AI → AI 改 → 你再发现新问题...
  Loop 模式：你下指令 → AI 写完自动检查 → 发现报错自己改 → 直到跑通为止

  设计和编码是分开的，为什么分开？因为 AI 的注意力有限——如果在一个对话里既设计又写代码，写到后面它就忘了前面的设计了，最后代码可能偏离方案。
  
  怎么加到你的流程里

  编码阶段的 prompt 改成这样：

  现在开始编码，按以下规则工作：

  ## 规则
  1. 每完成一个功能模块，自动运行测试/启动服务验证
  2. 如果报错，自己分析错误原因并修复，然后重新验证
  3. 重复以上循环，直到：
     - 服务成功启动，没有报错
     - 所有核心功能点击一遍都能正常工作
     - 连续 2 次循环没有新的问题出现（说明已经稳定了）
  4. 每次循环结束时，告诉我：
     - 修了什么
     - 还有什么问题
     - 是否已达到稳定状态

  现在开始第一阶段：搭建项目骨架，实现工单列表和详情页的基础 CRUD。

  实际效果

  普通模式你需要说 10 次话：
    你：开始写代码
    你：报错了，xxx
    你：还是不行，xxx
    你：样式乱了
    你：数据库连不上
    ...

  Loop 模式你说 1 次：
    你：[上面的 prompt]
    → AI 自己循环 5-8 轮，最后告诉你结果

  什么时候用 Loop，什么时候不用

  ┌──────────────────────────────┬──────────┬─────────────────────────────┐
  │             场景             │   模式   │            原因             │
  ├──────────────────────────────┼──────────┼─────────────────────────────┤
  │ 搭框架、写 CRUD              │ Loop     │ AI 自己能验证，不需要你介入 │
  ├──────────────────────────────┼──────────┼─────────────────────────────┤
  │ 做设计决策                   │ 手动     │ 需要你的判断                │
  ├──────────────────────────────┼──────────┼─────────────────────────────┤
  │ AI 连跑 3 轮修不好同一个问题 │ 手动介入 │ 说明方向错了，需要人来看看  │
  ├──────────────────────────────┼──────────┼─────────────────────────────┤
  │ 写系统 prompt / AI 分析逻辑  │ 手动     │ 需要人工判断质量            │
  └──────────────────────────────┴──────────┴─────────────────────────────┘

  核心原则：能自动验证的就 Loop（编译、测试、启动），需要人判断的就手动。
