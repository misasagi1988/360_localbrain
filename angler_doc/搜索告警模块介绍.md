# 搜索告警模块介绍

标签（空格分隔）： ANGLER_DESIGN

---

## 介绍
搜索告警是以日志为数据源，利用es的过滤聚合能力，批量定时查询es，对比聚合结果及规则告警触发条件，来生成告警的模块。
聚合方法支持count/distinct_count/sum/avg/max/min，支持自定义配置分组字段。
目前支持三种分析方式：

- 聚合分析
  
  在一定时间窗口内的统计结果值满足规则触发条件即可触发告警
  
- 同比分析

同比周期与同期数计算方法

  | 同比周期  | 时间窗口不超过1天时同期结果计算             | 时间窗口超过1天时同期结果计算                 |
  | --------- | ------------------------------------------- | --------------------------------------------- |
  | 昨天      | 与昨天同一时间段的统计结果做对比            | 无意义                                        |
  | 上周      | 计算上周每天同一时间段的统计结果，取平均    | 与上周同一时间段的统计结果做对比              |
  | 上个月    | 计算上个月每天同一时间段的统计结果，取平均  | 计算上个月每周同一时间段的统计结果，取平均    |
  | 自定义N天 | 计算前面N天每天同一时间段的统计结果，取平均 | 计算前面N天中每周同一时间段的统计结果，取平均 |
  
时间窗口配置
  
  | 单位  | 可选值             |
  | --------- | ------------------------------ |
  | 分钟      | 15/30/60            |
  | 小时      | 1/8/24    |
  | 天    | 1/7  |

触发条件：一定时间窗口内的统计结果的变化率满足条件，本期统计结果值满足条件(选填)

- 基线分析

  基线结果：基线时间窗口内以基线计算间隔做分隔后求平均，基线周期单位可配置为小时/天。
  
## 改造
### 2020年12月份
改造为达到时间窗口长度后再执行检查是否生成告警逻辑
### 2022年6月份信息安全部改造
问题：当前针对长时间的告警降噪、多目标关联等场景只能用检索分析规则才能不漏告警，随之而来就会面对告警实时报送，还是时间窗口结束再报告警的问题，而告警延迟他们不能接受的。从安全分析场景需求的角度来讲，不同的统计方法的告警时间确实应该是不一样的。
