## 2.0版本

### 威胁情报维度

威胁情报匹配由shuri模块来完成。shuri会消费kafka的实时日志数据，跟系统环境的威胁情报数据做匹配，生成威胁情报IP/Domain/URL匹配事件，回送到kafka，交给sae做分析。

sae的威胁情报匹配类规则不支持历史任务。

#### 在规则中使用威胁情报匹配

事件源可以使用威胁情报IP/Domain/URL匹配事件，过滤条件中可以使用match操作符指定匹配字段；如果事件源使用全局事件，必须在过滤条件中指定匹配字段。

威胁情报字段名称以"情报类型."开头，威胁情报维度相关字段仅可用于过滤条件、关联条件配置。

#### 情报类型

| 情报来源/类型 | IP_PORT | IP | Second-level domain | Fully Qualified Domain Name | URL  |
| :------------- | :---------- | :------------------ | :-------------------------- | :--- | :--- |
| 360 ioc情报    | IP_PORT     |      | SLD                 | FQDN                        | URL  |
| 用户自定义情报 | IP_PORT     | IP   | SLD                 | FQDN                        | URL  |

#### 情报匹配逻辑

shuri模块会分别提取日志的源地址/目的地址/域名/URL四个字段值，与当前环境中的情报数据做比较，如果发现匹配，则会生成威胁情报匹配事件，并将该威胁情报的具体信息放入ti_dimension字段，将原来的事件名称保存到orignal_event_name字段，ioc_matcher字段表明了具体的匹配字段。

**ip匹配**

- 源地址匹配：提取事件的源地址(src_address)，计算其sha256值并加密，与用户自定义的IP类型情报进行碰撞匹配，若命中且其alert字段值为true，则生成威胁情报IP匹配事件，ioc_matcher: src_address。
- 目的地址匹配：提取事件的目的地址(dst_address)、目的端口(dst_port)字段值做拼接：dst_address:dst_port，计算其sha256值并加密，与IOC情报中IP_PORT类或用户自定义的IP_PORT情报做匹配， 若命中且其alert字段值为true，生成威胁情报IP匹配事件，ioc_matcher: dst_address。

**域名匹配**

- 域名匹配： 提取事件的域名(domain_name)
- 二级域名匹配：提取其二级域名，计算二级域名sha256值并加密，与IOC情报中的SLD类情报或用户自定义的SLD类情报做匹配，若命中且alert字段的值为true，生成威胁情报Domain匹配事件，ioc_matcher: domain_name，该条日志域名匹配结束。
- 全域名匹配：若二级域名未命中，计算完整域名sha256值并加密，与IOC情报中的FQDN类情报或用户自定义的FQDN类情报做匹配，若命中且alert字段的值为true，生成威胁情报Domain匹配事件，ioc_matcher: domain_name。

**URL匹配**

- URL匹配：提取事件的URL(http_url_path)
- 完整URL匹配：计算URL的sha256值并加密，与IOC情报中的URL类情报或用户自定义的URL类情报做匹配，若命中且alert字段的值为true，生成威胁情报URL匹配事件，ioc_matcher: http_url_path，该条日志URL匹配结束。
- 若完整URL未匹配，提取URL中的host，若host为域名，则调用域名匹配逻辑，若命中，生成威胁情报Domain匹配事件，ioc_matcher: http_url_path；若host为ip，则提取URL中的ip和port调用IP目的地址匹配逻辑，若命中，生成威胁情报IP匹配事件，ioc_matcher: http_url_path。

源地址匹配、目的地址匹配、域名匹配、URL匹配都会做一次，最坏情况下，一笔日志可以生成四个威胁情报匹配事件。

### 资产维度

资产匹配由sae-core自己实现，sae会提取日志中的源地址或目的地址字段，与系统环境中的主机资产数据做关联匹配。

sae的资产匹配类规则支持历史任务。

#### 在规则中使用资产匹配

事件源没有限制。

字段名称以"资产.源地址."开头的用于使用日志的源地址与内部主机资产匹配；字段名称以"资产.目的地址."开头的用于使用日志的目的地址与内部主机资产匹配。资产维度相关字段仅可用于过滤条件、关联条件配置。

#### 资产匹配逻辑

如果规则的过滤条件或关联条件中使用了"资产.源地址."开头或"资产.目的地址."开头的字段，sae会根据规则配置的与资产关联的字段，得到系统环境中的相关资产信息，将资产数据丰富化到日志中，进行过滤。字段以"资产.源地址."开头，表明使用日志的源地址与系统环境中的资产数据做关联；字段以"资产.目的地址."开头，表明使用日志的目的地址与系统环境中的资产数据做关联。

资产数据丰富化到日志，会修改日志数据，<font color=red>多esper引擎处理时会导致并发问题</font>，这是2.0版本的一个bug，使用时需谨慎。

### 脆弱性维度

脆弱性匹配由shuri模块来完成。shuri会消费kafka的实时日志数据，跟系统环境的脆弱性数据做匹配，生成漏洞利用匹配事件，回送到kafka，交给sae做分析。

sae的脆弱性匹配类规则不支持历史任务。

#### 在规则中使用脆弱性匹配

事件源可以指定漏洞利用匹配事件。

脆弱性相关字段名称以"脆弱性类型."开头，脆弱性维度相关字段仅可用于过滤条件、关联条件配置。

#### 脆弱性匹配逻辑

shuri模块会提取日志中的目的地址字段值，与当前环境中的资产脆弱性数据做比较，如果发现匹配，则生成漏洞利用匹配事件，并将该资产脆弱性的具体信息放入vulnerability_dimension，将原来的事件名称保存到orignal_event_name字段。

匹配逻辑：提取日志的dst_address及vulnerability_id，与脆弱性数据的ip_address、cve_list/cncve_list/cnvd_list/cnnvd_list做比较，只要ip_address存在，且vulnerability_id的任意一个cve在4个list中存在，且该目的地址属于内置资产，则匹配生成漏洞利用匹配事件。



## 0130版本改动

### 威胁情报维度

#### 情报类型改动

增加IP信誉情报，情报类型有以下几种：

| 情报来源/类型  | IP_PORT | IP            | Second-level domain | Fully Qualified Domain Name | URL  |
| :------------- | :------ | ------------- | :------------------ | :-------------------------- | :--- |
| 360 ioc情报    | IP_PORT |               | SLD                 | FQDN                        | URL  |
| 用户自定义情报 | IP_PORT | IP            | SLD                 | FQDN                        | URL  |
| IP信誉情报     |         | IP_REPUTATION |                     |                             |      |

IP信誉情报不包含端口信息。

#### 情报匹配逻辑改动

因为增加了IP信誉情报，IP匹配逻辑有改动，shuri模块会依据IP情报类型对源地址、目的地址做分别匹配。域名匹配、URL匹配逻辑没有改动。

**IP情报匹配**

- 源地址匹配： 提取事件的源地址(src_address)，计算其sha256值并加密，与用户自定义的IP类型的情报及IP信誉情报进行碰撞匹配，若命中一种类型且其alert字段值为true，丰富化字段中类型字段为ti_type(数组格式)，生成威胁情报IP匹配事件，ioc_matcher: src_address；若两种类型都命中其alert为true，ti_type、tags字段合并，其余字段取两者的较大值，生成威胁情报IP匹配事件，ioc_matcher: src_address。
- 目的地址匹配： 提取事件的目的地址(dst_address)，计算其sha256值并加密，与用户自定义的IP类型的情报及IP信誉情报进行碰撞匹配，若命中一种类型且其alert字段值为true，丰富化字段中类型字段为ti_type(数组格式)，生成威胁情报IP匹配事件，ioc_matcher: dst_address；若两种类型都命中其alert为true，ti_type、tags字段合并，其余字段取两者的较大值，生成威胁情报IP匹配事件 ioc_matcher: dst_address。

**IP_PORT情报匹配**   

- 源地址匹配： 提取事件的源地址(src_address)、源端口(src_port)字段值做拼接：src_address:src_port，计算其sha256值并加密，与IOC情报中IP_PORT类或用户自定义的IP_PORT情报做匹配， 若命中一种类型且其alert字段值为true，丰富化字段中类型字段为ti_type(数组格式)，生成威胁情报IP匹配事件，ioc_matcher: src_address；若两种类型都命中其alert为true，ti_type、tags字段合并，其余字段取两者的较大值，生成威胁情报IP匹配事件 ioc_matcher: src_address。
- 目的地址匹配： 提取事件的目的地址(dst_address)、目的端口(dst_port)字段值做拼接：dst_address:dst_port，计算其sha256值并加密，与IOC情报中IP_PORT类或用户自定义的IP_PORT情报做匹配， 若命中一种类型且其alert字段值为true，丰富化字段中类型字段为ti_type(数组格式)，生成威胁情报IP匹配事件，ioc_matcher: dst_address；若两种类型都命中其alert为true，ti_type、tags字段合并，其余字段取两者的较大值，生成威胁情报IP匹配事件 ioc_matcher: dst_address 。

源地址匹配、目的地址匹配既会做IP情报匹配，也会做IP_PORT情报匹配，最坏情况下，一笔日志可以生成六个威胁情报匹配事件。

### 资产维度

#### 规则引用字段改动

增加主机ip与资产数据的关联，字段名称以” 源地址.资产.”、”目的地址.资产.”、”主机IP.资产.”开头。

#### 资产匹配逻辑改动

修复丰富化数据导致的并发问题，修改资产匹配逻辑，自定义函数依据字段获取对应资产数据，实现过滤匹配。

### 脆弱性维度

#### 规则引用字段改动

移除漏洞利用匹配事件，不再依赖shuri模块，由sae自己实现脆弱性数据的过滤匹配。

增加源地址、主机ip与脆弱性数据的关联，字段名称以” 源地址.资产.”、”目的地址.资产.”、”主机IP.资产.”开头。

#### 脆弱性匹配逻辑改动

不再依赖shuri模块，由sae自己实现脆弱性数据的过滤匹配，匹配时不检查资产数据。

由于一个IP地址可对应多条脆弱性数据记录，需要把这些脆弱性数据以IP地址做分组，对分组内数据的每个字段做合并后再做匹配。

目前，脆弱性表达式不支持左右操作数都是脆弱性相关字段，只允许单个操作数是脆弱性相关字段。

| 操作符    | 操作数           | 匹配逻辑                                                     |
| --------- | ---------------- | ------------------------------------------------------------ |
| =         | 左右操作数都支持 | 该IP地址对应的多个脆弱性数据中，有一条数据该字段的值与待匹配值相同，即匹配 |
| !=        | 左右操作数都支持 | 该IP地址对应的多个脆弱性数据中，有一条数据该字段的值与待匹配值不同，即匹配 |
| >         | 左右操作数都支持 | 该IP地址对应的多个脆弱性数据中，有一条数据该字段的值大于待匹配值，即匹配 |
| >=        | 左右操作数都支持 | 该IP地址对应的多个脆弱性数据中，有一条数据该字段的值大于等于待匹配值，即匹配 |
| <         | 左右操作数都支持 | 该IP地址对应的多个脆弱性数据中，有一条数据该字段的值小于待匹配值，即匹配 |
| <=        | 左右操作数都支持 | 该IP地址对应的多个脆弱性数据中，有一条数据该字段的值小于等于待匹配值，即匹配 |
| exist     | 左操作数         | 该IP地址对应的多个脆弱性数据中，有一条数据存在该字段，即匹配 |
| not_exist | 左操作数         | 该IP地址对应的多个脆弱性数据中，所有数据都不存在该字段，即匹配 |
| like      | 左右操作数都支持 | 该IP地址对应的多个脆弱性数据中，有一条数据该字段的值包含待匹配字符串，即匹配。只支持字符串类型字段。 |
| rlike     | 左右操作数都支持 | 该IP地址对应的多个脆弱性数据中，有一条数据该字段的值匹配正则表达式，即匹配。只支持字符串类型字段。 |
| in        | 左操作数         | 左操作数为IP: 脆弱性数据该字段合集中，有一个值属于右值IP区间，即匹配。<br>右操作数为数组: 脆弱性数据该字段合集与该数组有交集，即匹配。 |
| belong    | 左操作数         | 该IP地址对应的多个脆弱性数据中，有一条数据该字段的值匹配对应信息组内容，即匹配 |
| contain   | 左右操作数都支持 | 左操作数为脆弱性数据: 右操作数为非数组类型，字段合集包含待匹配值，即匹配。<br>右操作数为脆弱性数据: 左操作数为数组类型，两个数组有交集，即匹配。 |
| match     | 不支持           | 不支持                                                       |
