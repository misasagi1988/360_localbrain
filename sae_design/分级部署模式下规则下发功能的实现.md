# 分级部署模式下规则下发功能的实现

标签（空格分隔）： SAE_DESIGN

---

### 需求描述

多级部署模式下，上级节点可以将本节点的某些SAE规则下发到下级节点，并在下级节点引擎中加载运行，生成告警。
目前，父节点将规则导出后交由子节点导入即可实现该功能，但如果子节点数量过多，导出导入的运维太复杂，而且如果发现规则有问题，想禁用，又得一个一个系统去改，操作相当繁琐。
因此想实现一个功能：多级部署模式下，如果父节点允许规则下发，则自动将该规则下发到其所有子节点，交由子节点加载运行。规则修改、删除后也自动同步到子节点。

### 实现思路

**页面改动**

- 多级部署模式下，父节点规则配置页面增加是否允许下发规则的配置选项，如果用户选择下发规则，则将规则下发到所有的子节点。
- 针对父节点下发的规则，子节点有只读权限。

**分析模块实现逻辑**

- 与全局关联分析功能类似，父节点使用本地kafka server，通过message-bus将发出规则配置，topic定义为qihoosoc-distribution-sae-rule。子节点消费父节点kafka server topic数据，将规则配置写入本地db(这里与全局关联分析类似，规则信息发往本节点kafka，原因，如果发到子节点kafka，存在多个子节点的情况，需要创建多个consumer，无意义)。
- 如果子节点存在下级节点，需要将规则配置信息同步到下级各个子节点。
- 规则更新时，父节点发出规则更新消息，子节点消费该消息，根据消息的指定动作做相应处理。
- 规则创建/编辑/更新状态/删除时，父子节点都需要根据下发使能状态做不同处理(目前想到的)：
  - add:
    父节点: 如果用户配置规则下发使能，则需将规则下发到子节点
    子节点: 收到父节点规则，添加该规则到本节点，规则只读。
  - edit:
    父节点: 
    distributeEnabled=true->distributeEnabled=false, delete消息
    distributeEnabled=false->distributeEnabled=true, add消息
    distributeEnabled=true->distributeEnabled=true, update消息
    distributeEnabled=false->distributeEnabled=false, ignore
    子节点: 根据不同消息做不同处理。
  - update status:
    父节点: 提取使能下发的规则，发送update status消息
    子节点: 根据消息加载或移除规则。
  - delete:
    父节点: 提取使能下发的规则，发送delete消息
    子节点: 根据消息删除规则。
- 下发的规则id和name都不变，如果子节点id有冲突，则覆盖子节点规则；如果名称有冲突，则重命名该规则。
- 为了与全局关联分析功能区分开来，防止因node_chain存在导致子节点无法正常加载(子节点根据node_chain字段判断是本节点规则or父节点规则，父节点规则不加载到引擎，只做全局关联分析)，下发的规则不带node_chain字段；鉴于规则还要同步下发到下级子节点，该规则在子节点中不使能全局关联分析。
- 规则导出时，只允许导出本地规则，且不使能下发，不使能全局关联分析。

### 2.2版本双子星模式改动
- 支持下发节点选择：可以选择下发到全部子节点，一个或多个下级节点。节点列表会列出全部子节点。
  - 下发到全部节点：下发节点列表中包含特殊字符串标识：TO-ALL
- 支持在规则界面快速过滤下发到下级节点的规则及上级下发的规则。
- 规则导入影响
  - 内容包导入内置规则，如果规则下发，则modified标记位为true，不会通过内容包更新
  - 页面导入：依据用户选择而定
- cps_rollback影响
  - 内置规则，如果规则下发，则modified标记位为true，不会删除
  - 非内置规则，不影响

### 注意事项

 - 流式分析规则和检索分析规则都支持该功能。
 - 子节点如果存在下级节点，需要将规则配置信息同步到下级各个子节点。
 - 子节点只允许查看该规则，无编辑/状态更新/删除/导出权限。
 - 有新的节点注册时，不支持父节点将已经使能下发的所有规则下发给新注册的节点。
 - 节点注销时，不支持子节点删除所有父节点下发的规则。
 - 升级时注意，升级到2.2版本，所有下发规则的"distributeTo"=["TO-ALL"]，标识该规则下发至所有子节点。
