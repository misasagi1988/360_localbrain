## 疑问

 检索分析规则是否需要支持该功能？-- Yes
 子节点是否需要加载使能全局关联分析的规则？本脑当下是不支持的。-- No, 子节点不加载使能全局关联分析的规则，它只需要把满足条件的数据发到上级节点
检索分析规则相关：
不使能的全局关联分析的：父节点只能查询本节点的数据，但父节点无数据，这种规则对父节点无意义；子节点是否加载？
使能的全局关联分析的：父节点可查询所有子节点的数据，是选择查询哪些子节点的，还是所有节点都查？

内置规则的问题，无法编辑

## 思考

sae: 
主，加载global规则，消费从节点发过来的数据，输出告警；加载非global规则，只能消费本节点数据，但本节点无数据，这些规则无法生成告警。
从，不能加载global规则；只加载非global规则，消费本节点数据，输出告警。加载全局filter规则，输出满足条件数据，给主节点消费。
现状：从节点生成告警全部是非global规则的；主节点生成告警全部是global规则的。


angler，主从只有一个
不论是否开启global开关，规则都要被加载，因此，该开关有何意义？
是否需要配置选择过滤哪些个节点的数据，去匹配过滤分析？

规则页面配置改动：
去除全局关联分析开关，去除下发规则开关，”下发规则“功能移除
全局关联分析功能改动，添加规则运行节点配置选项，可以配置四个节点，全局关联分析开关


暂时讨论结果：
流式要满足下面3个模式  
A. 默认，规则在4个节点生效（只分析单一数据流）  
B. 指定节点分析，规则在哪些节点生效（只分析单一数据流）—— 要支持多选节点  
C. 全局分析，规则在所有节点的数据上（分析多数据流）—— 先推客户不选全局的域，但是如果客户要也可以做  
检索要满足下面2个模式  
A. 单选某个节点，只分析这个节点的数据  
B. 多选多个节点，跨这些节点数据进行分析

## 开发
### sae-monitor
规则增加analysisMode，分析模式: 0=默认模式 1=指定节点模式 2=全局分析模式
targetNodes，目标节点列表，指定节点模式时生效
增加db更新脚本。
其他不变。

### sae-core
配置文件增加节点相关配置，用于判断节点类型：
```yaml
analysis:
  nodes:
    current-node: ""          # 当前节点名称（必填）
    parent-nodes: []          # 上级节点列表
    child-nodes: []           # 下级节点列表
```
配置文件的kafka.global-event-producer改动，父节点bootstrap.servers留空，子节点bootstrap.servers配置为父节点kafka server
上级节点加载所有规则；生成全局大规则。
下级节点加载默认规则及指定自己加载的规则；加载全局大规则，匹配后发到上级kafka server。

### angler
angler 引擎只部署在上级节点上，可在请求 header 中添加 `cross_cluster` 字段指定下级节点，值为**节点ID**，多个逗号分隔
选一个节点多个节点都是选，只是请求hql时的cross_cluster的值不一样。


### 前端
规则下发功能隐藏，全局关联分析的“指定节点分析”就是规则下发的效果，没必要再去做是否使能规则下发及规则下发节点选择。
全局关联分析配置改造，分三个选项，第二个选项可以借用规则下发节点选择器的功能。
