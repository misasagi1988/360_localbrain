 规则评估处理流程参考[规则评估流程.pdf](规则评估流程.pdf)
 存在的问题：
 1. 代码逻辑比较乱，中间生成了部分无用文件，有些数据结构比较庞大，可以删除，有很多可以优化的地方
 2. 获取事件名称的数据链接口不支持分页查询，如果一个事件名称关联的解析规则链有上万个，response body太大，可能会有问题，建议增加分页支持。接口数据的准确性需找DV进一步确认。以”登录事件“的"VPN_深信服_SSL系列_JSON_入口规则"->"VPN_深信服_SSL系列_系统日志_JSON_子规则"为例，event_type不准确，不存在"未知用户登录"。
 3. 在解析规则的解析结果展示中，如果没有映射成标准字段，该字段会不会被解析并输出到日志中？通过字段的解析结果的mapping映射、默认值，是否可以得到该规则的这个字段的所有可能值，这些需找DV进一步确认
 6. 规则评估代码逻辑存在问题：
     - 只用了过滤条件的字段做评估，规则的分组字段、关联条件没有参与评估，导致评估结果不准确
     - 不支持inmap/not_inmap操作符
     - 评估时，如果存在多个问题，后面的问题会把前面的问题覆盖掉
     - 数据结构问题：规则的数据结构：sae_rules[rule_name][event_name][满足条件的入口规则]=dict(), 对于使用多个事件的sae规则，比如一个follow-by规则，如果规则引用了两个事件名称相同的事件，如果A事件有解析规则链满足了条件，则不再评估B事件，导致评估结果不准确----这个问题我想错了，event_name不是事件名称，而是事件别名；解析规则链的数据结构：parse_tree[event_name][uuid][问题][sae_rule]=dict()，同样是一个follow-by规则引用了两个事件名称相同的事件，如果AB两个事件都不满足条件，B又会把A的问题给覆盖掉
     - 内部事件评估逻辑问题：内部事件的评估结果直接使用了生成内部事件的规则的评估结果，考虑到生成内部事件的规则可能有多个，内部事件也可以有过滤条件，不建议这么做

### 评估数据源范围

### 可评估规则分类

#### 通用|专用

- 通用规则：适用针1-N 个设备类型的规则，绝大部分规则都是通用规则
- 专用规则：使用了数据源、数据源厂商、数据源产品进行过滤的规则，进针对特定厂商、产品生效的规则

#### 单一设备类型|多设备类型

- 单一设备类型：规则仅引用1 组（1-n个， 存在任意 1 个即满足）设备类型的日志，绝大部分规则都是单一设备类型。
- 多设备类型：规则引用 2 组不同类型设备日志，例如 follow-by 规则， 事件 a 为网络类，事件 b 为终端类，则规则同时需要网络、终端类型设备接入后才可能生效。

### 针对规则的评估逻辑

#### 通用规则

1. 根据规则对应的设备类型检索当前系统接入的所有数据源，获取评估数据源
2. 依次评估每个数据源是否满足需求
3. 检索一段时间内所有该规则对应的告警，获取实际出发告警的数据源列表

#### 专用规则

基本流程同通用规则，增加对三个字段的判断，判断是评估数据源







### 使用接口
获取数据源接口：
```
request: GET /__api/dv/datasource
params:
start/limit: 分页参数，必须
order/orderBy: 排序字段，非必须
search: 过滤条件，非必须，name_not_contain-数据源名称不包含，demo: {"name_not_contain":"_360XDR_","xdrId":null,"organization":"/1"}，过滤三方数据源
response:
{
	"statusCode": 0,
	"messages": [],
	"data": {
		"start": 0,
		"limit": 20,
		"totalCount": 3,
		"currentPage": 1,
		"totalPage": 1,
		"result": [] //数据源列表
	}
}
```

获取解析规则接口：
```
request: GET /__api/dv/resolver
params:
start/limit: 分页参数，必须
order/orderBy: 排序字段，非必须
search: 过滤条件，非必须
response:
{
	"statusCode": 0,
	"messages": [],
	"data": {
		"start": 0,
		"limit": 20,
		"totalCount": 3,
		"currentPage": 1,
		"totalPage": 1,
		"result": [//解析规则列表
		    {
		    "name": "", //名称
		    "deleted": 0, //是否被删除
		    "subResolver": 0, //是否只作为子规则
		    "simEntities": [], //推荐对象
		    "filter": {}, //数据过滤
		    "normalize": [], //解析结果
		    "properties": [] //预览结果
		    }
		] 
	}
}
```

解析规则链接口：
```
request: POST /__api/dv/resolver/list?eventName={}
response:
{
	"statusCode": 0,
	"messages": [],
	"data": {
		"resolvers": [
		{
		    "resolverName": "",
		    "resolverNameList": [], //解析链里关联的所有规则名称列表
		    "deviceTypeList": [], //解析链里的规则配置的设备类型列表
            "keyword": {}, //一些映射字段的值列表
            "fields": [] //可解析出来本脑字段
		}
		]
	}
}
```
attention: 
keyword只限定下面字段，在解析规则链里赋值的值 
threat_type,result,event_operation_type,log_vendor,event_action,attack_result,product,data_source,event_level,event_type,log_type,os,login_app,threat_info,event_description,event_title,threat_name
